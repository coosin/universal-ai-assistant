# 开发复盘与收尾（OpenClaw + CLIProxyAPI + Telegram）

本文总结本仓库从“从零跑起来”到“可在终端/Telegram 对话”的完整过程、踩坑与处理方式，并给出最终的收尾清单。

---

## 一、目标与架构

- **目标**：在 WSL2/Ubuntu 上把个人 AI 助手跑起来，支持：
  - 终端对话（`openclaw agent`）
  - Telegram 对话（可选）
  - 多模型/多 Provider 的自动 fallback（遇到计费不足、网络失败时自动切换）
- **核心组件**：
  - **OpenClaw Gateway**：负责路由、会话、工具、fallback
  - **CLIProxyAPI**：把 Claude Code / Gemini / Codex 等账号能力封装为 OpenAI 兼容 API（`/v1/...`）
  - **可选 Web 面板**：状态/健康检查/配置校验

---

## 二、关键里程碑与问题处理

### 1）环境准备（WSL2）

- **问题**：Node.js / Docker 未安装或版本不满足
- **处理**：文档化并固化到 `docs/正式跑起来.md`，并提供 `install.sh` 一键安装/拷贝示例配置

### 2）全局 npm 权限（EACCES）

- **问题**：`npm install -g openclaw` 因 `/usr/lib/node_modules` 写权限失败
- **处理**：`install.sh` 里先尝试普通安装，失败自动提示并使用 `sudo npm install -g openclaw@latest`

### 3）CLIProxyAPI 镜像与端口

- **问题**：某些镜像源拉取失败（权限/denied），端口与面板访问受限
- **处理**：
  - 统一 Docker Compose 启动（`docker-compose.yml`）
  - 管理面板要求 `remote-management.secret-key` 非空、`allow-remote: true`
  - 主端口统一为 `8317`，OpenClaw 通过 `http://localhost:8317/v1` 调用

### 4）CLIProxyAPI YAML 易错点

- **问题**：YAML 重复 key、换行/缩进导致容器反复重启，`proxy-url` 格式不合法等
- **处理**：
  - 维护可工作的示例：`config/cliproxyapi.yaml.example`
  - 在 `docs/正式跑起来.md` 中强调“先跑面板拿 API Key，再填 OpenClaw”

### 5）OpenClaw 配置 schema 迁移

- **问题**：旧 schema 导致 `Config invalid / Legacy config keys`，`gateway.bind` 不接受 `127.0.0.1`
- **处理**：
  - 统一新 schema：`agents.defaults` + `models.providers`
  - `gateway.mode: local` + `gateway.bind: loopback`
  - 必要时 `openclaw doctor --fix` 一键修复

### 6）模型路由与 fallback（“计费不足”）

- **现象**：CLIProxy（Claude）或 OpenRouter 返回 “余额不足/Insufficient credits”
- **根因**：
  - CLIProxy 背后账号（Anthropic/Claude）余额不足时会直接返回错误
  - OpenRouter 的 `openrouter/free` 虽是免费路由，但**账号/API Key 仍可能需要可用 credits**（否则 402）
- **处理策略**：
  - 把“更可能可用的模型”放在 **primary**（例如 CLIProxy 的 Gemini/Antigravity、或 Codex）
  - 把“可能会计费不足的”放在 **fallback**（Claude/OpenRouter）
  - 文档化：`docs/OpenClaw与CLIProxy调用逻辑.md`

### 7）Telegram 通道、配对与国内网络

- **问题 A**：`OpenClaw: access not configured` / pairing code
- **处理**：首次私聊 Bot 后，在 WSL 执行：
  - `openclaw pairing approve telegram <PairingCode>`
- **问题 B**：`TypeError: fetch failed`（能收到消息但不回复）
- **根因**：WSL 内访问 Telegram/OpenRouter 失败（国内网络/代理不通）
- **处理**：
  - 增加 `.env` 加载：`start.sh` 会自动 `source .env`
  - 提供 `.env.example` 指引（WSL2 NAT 下需用 Windows 宿主机 IP，而不是 localhost）
  - 健康检查增加对 Telegram/OpenRouter 连通性提示：`scripts/health_check.sh`

## 四、最终收尾清单（建议按顺序核对）

1. **启动**：`./start.sh`（保持终端常开）
2. **健康检查**：`./scripts/doctor.sh`（看是否提示 Telegram/OpenRouter 需要代理）
3. **终端对话**：`openclaw agent --message "你好"`
4. **Telegram**（可选）：
   - 配好 `channels.telegram.botToken`
   - 私聊 Bot 获取 Pairing code
   - `openclaw pairing approve telegram <code>`
5. **密钥安全**：
   - 不要把 `.env`、`~/.openclaw/openclaw.json`、`~/.cliproxyapi/config/config.yaml` 提交到仓库
   - 如果 Token/Key 在聊天或截图中暴露，建议立刻轮换
